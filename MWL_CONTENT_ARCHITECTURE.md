# Math With Love — архитектура контента и знаний

Документ фиксирует архитектурную модель, согласованную в обсуждении (20 июня 2026).
Обновлён: 29 июня 2026 — §11 навигационный путь синхронизирован с /courses (три уровня), статусы тем задокументированы.
Обновлён: 05 июля 2026 — §8 добавлен пятый экран конструктора «Управление шейлоном»: иерархия шейлон→тема→модуль, архитектура shared атомов, дублирование структуры. §9 добавлен открытый вопрос по Supabase-схеме для шейлонов. §10 обновлён.
Статус: **рабочее предположение**, требует верификации по мере реализации.

---

## 1. Цель системы

Главная цель — самообучение на основе интереса студента, в игровом формате,
максимально модулярное. Жёсткая линейная программа — это анти-цель: система
должна позволять студенту двигаться по контенту не только в заранее
зафиксированном порядке, но и исследовать его, насколько позволяют
зависимости между единицами знания.

Источник истины по содержанию — Багрут 572 (и частично 571), 5 яхидот.

---

## 2. Базовая единица — «атом знания»

Атом — минимальная самодостаточная единица контента. Не вложен физически
ни в какую структуру (Тема/Модуль и т.п. — это теги/группировки, а не контейнеры,
см. раздел 4).

Внутренняя структура атома (фиксированный порядок шагов внутри одного атома):

1. **Теория** — подача нового материала + динамичная графика
2. **Проверка понимания** — короткий тест сразу после теории
3. **Упражнения** — последовательная система закрепления
4. **Тест атома** — итоговая проверка закрепления именно этой единицы

Поля с уравнениями вводятся через **MathLive** (`<math-field>`) — WYSIWYG,
без видимого LaTeX-кода. Числовые поля проверяются обычным сравнением.
Ответы на формулы проверяются ИИ (механизм не спроектирован — см. §9).

---

## 3. Контрольные точки (checkpoints)

Узлы графа второго типа (наравне с атомами).

| Контрольная точка | Когда возникает | Что проверяет |
|---|---|---|
| Проверка понимания | Внутри атома, после теории | Только что введённые понятия (часть атома, не отдельный узел) |
| Тест атома | В конце атома | Закрепление конкретного атома (часть атома) |
| Контрольная по группе | После прохождения ВСЕХ атомов группы | Синтез/применение знаний по группе |
| Экзамен по крупной группе | После прохождения нескольких групп | Несколько заданий на разные темы |
| Итоговый экзамен (571/572) | Финал | По «схеме экзамена» — см. раздел 6 |

**Правило КТ группы (зафиксировано июнь 2026):** контрольная точка группы открывается
автоматически, когда пройдены все атомы группы. Порядок прохождения не важен.
Автор не настраивает состав КТ вручную — это всегда «все атомы группы».

«Проверка понимания» и «Тест атома» — внутренние шаги атома, не отдельные узлы.
«Контрольная по группе» и «Экзамен по крупной группе» — самостоятельные узлы графа.

## 4. Группировки — теги, не контейнеры

**Важно:** уровни группировки реализуются как **теги/фильтры поверх плоского
графа атомов**, а не как структурные контейнеры.

Один атом может относиться к нескольким группам одновременно (many-to-many).

Иерархия реализована через поле `parentId` на каждой группе (тег ссылается на
другой тег) — не через физическую вложенность. Глубина произвольная.

```
Блок программы Багрута
  └─ Группа верхнего уровня (пример: "Окружность")
       └─ Подгруппа (пример: "Общее уравнение окружности")
            └─ Атомы, связанные графом пререквизитов
```

Группа верхнего уровня закрывается «Контрольной по группе»;
блок программы — «Экзаменом по крупной группе»;
итог — Итоговый экзамен 571/572.

**Черновик атома** (зафиксировано июнь 2026): в Конструкторе групп автор может
создать заглушку-атом (только название, без содержимого) для планирования структуры
курса до наполнения. Черновик хранит `{id, title, titleHe, status:'draft', groups:[...]}`.
Становится полным атомом при открытии в Редакторе атома. Порядок атомов внутри
группы (`order`) используется только для отображения в конструкторе — реальная
последовательность обучения задаётся графом пререквизитов (раздел 5).

## 5. Граф пререквизитов

Каждый атом и каждая контрольная точка — узел одного графа.

Связь-пререквизит двух видов:
1. **Точечная** — атом Б требует именно атом А. Визуал: сплошная cyan линия.
2. **Через контрольную точку** — атом В требует, чтобы была сдана КТ группы Х.
   Визуал: пунктирная линия (slate-цвет).

Граф пререквизитов **статичен** (задаётся автором). Динамической является его
проекция для конкретного студента (раздел 7).

## 6. Примеры вопросников (экзаменационная схема)

Каждый экзамен Багрут (571, 572, 471, 472…) — отдельный **тип вопросника**.
Внутри типа — произвольное количество **экземпляров (сессий)**: конкретные
варианты экзамена, авторски названные, без привязки к дате.

Структура одного экземпляра:
- Метаданные: длительность, баллов за вопрос, количество вопросов для ответа
- Инструкции: свободный текст (не автогенерация)
- Разделы (פרקים): название + описание тематики + список вопросов
- Вопросы: текст (с inline-формулами и/или изображением) + баллы
- Правила выбора: «ответить на K из N», с ограничениями по разделам (поддержка OR-группировки)

«Примеры вопросников» — отдельная редактируемая сущность (не зашита в код).
Редактор — четвёртый экран конструктора контента (§8.4), **макет готов**.

**Ключевые решения (июнь 2026):**
- Экзамены **не привязаны к датам** — только к материалу
- Правила выбора влияют исключительно на валидацию при сдаче — не на видимость вопросов
- Чем больше экземпляров накоплено — тем лучше для практики студентов

## 7. Лабиринт = динамическая проекция графа

Лабиринт студента — не сам граф, а его проекция, отфильтрованная по прогрессу:
- пройдено → открыты следующие по графу
- заблокировано → пририквизиты не выполнены

Движок v17 (`s.segs`, `s.shortcutsUsed`, etc.) — **сохраняется без изменений**.
Поверх него нужен тонкий слой навигации: принимает `(граф, прогресс студента)`,
вычисляет видимые/заблокированные узлы. Не реализовано, не блокирует конструктор.

## 8. Конструктор контента — сущности для редактирования

1. **Редактор атома** — теория + графика, проверка понимания, упражнения, тест атома.
   Статус: **макет готов** (desktop + tablet). MathLive WYSIWYG. Двуязычный (HE/RU).
2. **Карта графа** — визуализация атомов и КТ со связями-пририквизитами.
   Статус: **макет готов** (desktop, оба theme). Force-directed SVG, pan/zoom/drag.
3. **Конструктор групп** — дерево групп/подгрупп + список атомов/черновиков + КТ-статус.
   Статус: **макет готов** (desktop, оба theme). Drag-and-drop, batch-перевод.
4. **Примеры вопросников** — двухуровневый редактор (тип шаалона → сессии), разделы,
   вопросы с формулами и изображениями, правила выбора, превью.
   Статус: **макет готов** (`mwl_exam_schema_v2.html`, desktop, оба theme, RU/HE).

Доступ: авторизованная оболочка (header + sidebar). Точка входа — пункт «Контент»
в сайдбаре с подменю (Карта графа / Редактор атома / Группы / Примеры вопросников / Шейлоны).

**Пятый экран конструктора — «Управление шейлоном» (зафиксировано июль 2026):**

Шейлон — верхний уровень иерархии контента: шейлон → темы → модули → атомы.
До появления этого экрана у автора не было инструмента для создания и управления
структурой курса на уровнях шейлон/тема/модуль.

**Иерархия (сверху вниз):**
```
Шейлон (например: «35572 — 5 яхидот»)
  └─ Тема (например: «Векторы»)
       └─ Модуль (например: «Основы векторов»)
            └─ Атомы (управляются в Карте графа и Редакторе атома)
```

**Ключевые архитектурные решения:**
- Атомы **глобальные** — не принадлежат конкретному шейлону или теме.
  Модули ссылаются на атомы через `prerequisite_edges`, не владеют ими.
- Дублирование шейлона = копия структуры (шейлон + темы + модули) + **shared ссылки на те же атомы**.
  Контент атомов не дублируется — изменение атома отражается во всех шейлонах, ссылающихся на него.
- Порядок тем и модулей хранится через поле `order` (integer) — **открытый вопрос по схеме Supabase**.
- Порядок может меняться при изменении программы Багрута — редактируется автором в этом экране.

**Функциональность экрана:**
- Дерево: шейлон → темы → модули (левая панель)
- Создание: кнопка «+» рядом с родительским узлом в дереве
- Редактирование названия: двойной клик → inline edit активного языка; detail panel справа → одно поле (активный язык)
- Переупорядочивание: drag-and-drop + кнопки ↑↓ на каждом уровне
- Создание/переименование/дублирование шейлона — кнопки в ctrl-bar
- Переход в редактор: клик на модуль → всплывающее окно (Открыть редактор атомов / Открыть граф / Отмена)
- Авто-перевод RU↔HE: запускается автоматически при нажатии «Сохранить» (через FastAPI backend)

**Detail panel:** одно поле названия (активный язык: `titleRu` или `titleHe`).
Описание шейлона: `descRu` / `descHe` — раздельные поля, показывается активный язык.
При переключении RU↔HE поля переключаются на соответствующий язык.
Batch-translate убран — перевод происходит автоматически при сохранении.

**Save + авто-перевод:** Save → toast "Сохранено ✓" → кнопка в shimmer "Переводим..." →
FastAPI вызывает Anthropic API → результат в `titleHe`/`descHe` или `titleRu` →
toast "Переведено ✓" → кнопка деактивируется. При ошибке: toast + кнопка "↺ Повторить".

**Статус: мокап готов** (`mwl_shalon_manager.html`, desktop only, оба theme, RU/HE, RTL).

**⚠️ Инженерный спайк (обязателен перед Фазой 8):**
Поля смешанного ввода (`contenteditable` + inline `<math-field>` + RTL bidi)
требуют отдельного spike в реальном Next.js окружении. Виртуальная клавиатура
MathLive пробивается через CSS; навигация по placeholder-слотам внутри shadow DOM
не проверена в браузере. Это техническая задача, не дизайнерская.

## 11. Структура темы «Векторы» (Багрут 572)

Зафиксировано июнь 2026. Первая реализованная тема платформы.

### Навигационный путь

```
/courses
  → карточка вопросника (35571 / 35572)
    → клик по теме «Векторы» (статус: ready)
    → попап силлабуса: список модулей + подтемы + кол-во уроков/тестов/экзаменов
      → кнопка «Начать» / «Продолжить»
      → /courses/vectors        — список модулей темы
        → клик на модуль
        → /courses/vectors/[moduleId]  — лабиринт атомов модуля (движок v17)
          → клик на атом
          → /lesson/[atomId]
```

Статусы темы на уровне 1:
- **ready** — попап с полным силлабусом + кнопка перехода
- **in_work** — попап «В работе» + иконка рабочего, без кнопки перехода
- **locked** — отображается с реальным названием + замок, клик недоступен

Все попапы закрываются по: кнопке ×, клику вне попапа, клавише Esc.

### Ключевое свойство темы

В Багруте по теме «Векторы» — **одна синтетическая задача**, которая требует
знания сразу нескольких модулей. Следствия:

1. Граф пририквизитов **пересекает границы модулей** — атом из Модуля 2 может
   быть пририквизитом атома из Модуля 5. Модули — это теги/фильтры, не барьеры.
2. КТ темы = **мини-экзамен** с синтетической задачей типа Багрут.
   Это отдельная сущность, не просто сумма КТ модулей.
3. Стереометрия **не выделена в отдельный поток** — она пронизывает все модули
   как обязательный контекст применения.

### Модульная структура

| Модуль | Название | КТ | Примечание |
|---|---|---|---|
| 0 | Стереометрия — введение | Нет | Вводный, короткий. КТ нет — достаточно тестов внутри атомов |
| 1 | Основы векторов | Да | Геометрический + алгебраический вектор, стереометрия |
| 2 | Скалярное произведение | Да | Геометрическое + алгебраическое, угол, стереометрия |
| 3 | Прямые и плоскости | Да | Явное + параметрическое, переходы, взаимосвязи, стереометрия |
| 4 | Расстояния | Да | Точка/прямая/плоскость, параллельные объекты, стереометрия |
| 5 | Углы | Да | Прямая-прямая, плоскость-плоскость, прямая-плоскость, стереометрия |
| — | КТ темы «Векторы» | — | Синтетическая задача типа Багрут |

### Детальный состав модулей

**Модуль 0: Стереометрия — введение** *(без КТ)*
- Точка, прямая, плоскость в пространстве
- Основные фигуры: куб, параллелепипед, призма, пирамида — свойства
- Параллельность и перпендикулярность в пространстве

**Модуль 1: Основы векторов** *(с КТ)*
- Понятие вектора, длина, направление
- Геометрические операции (сложение, вычитание, умножение на число)
- Координатная форма (алгебраический вектор)
- Применение в стереометрических фигурах

**Модуль 2: Скалярное произведение** *(с КТ)*
- Геометрическое определение (через угол)
- Алгебраическое определение (через координаты)
- Угол между векторами
- Применение в стереометрии (диагонали куба, рёбра пирамиды)

**Модуль 3: Прямые и плоскости в пространстве** *(с КТ)*
- Уравнение прямой (явное и параметрическое)
- Уравнение плоскости (явное и параметрическое)
- Переходы между формами
- Взаимосвязи (параллельность, перпендикулярность)
- Стереометрия: прямые и плоскости в телах

**Модуль 4: Расстояния** *(с КТ)*
- Расстояние между точками в пространстве
- Расстояние от точки до прямой / плоскости
- Расстояние между параллельными прямыми / плоскостями
- Стереометрия: высоты, апофемы, диагонали тел

**Модуль 5: Углы** *(с КТ)*
- Угол между двумя прямыми
- Угол между двумя плоскостями
- Угол между прямой и плоскостью
- Стереометрия: двугранные и трёхгранные углы в телах

### Статус атомов

Все атомы темы «Векторы» на данном этапе — **черновики** (только названия,
без содержимого). Наполнение начнётся после завершения проектирования среды
(Лаборатория Three.js / R3F) и принятия решения о рефакторинге.

### Карточка на странице /courses

- Статус: **ready** (единственная разработанная тема)
- Прогресс: показывается по модулям (N из 6 модулей завершено)
- Все остальные темы: **locked** или **in_work** — отображаются с реальными названиями

## 9. Открытые вопросы

- **Инженерный спайк: rich text + формулы.** `contenteditable` + inline MathLive +
  RTL bidi — известная сложная задача. Решать в Next.js до Фазы 8, не в HTML-макетах.
- ~~**Финальные названия уровней.**~~ **Зафиксировано (июнь 2026):** Тема = группировка верхнего уровня по вопроснику (/topics). Модуль = группа верхнего уровня внутри темы (группа с `parentId` = id темы). Атом = базовая единица. «Блок» — не используется.
- **Динамичная графика.** Механизм привязки Three.js-сцены к атому не спроектирован.
  В макете — только заглушка с переключателем типа (Вектор/Функция/3D/Плоскость).
- **Проверка ответов ИИ.** Промпт, формат сравнения эквивалентных форм формулы —
  бэкенд-логика, не UI. Не спроектирована.
- **Supabase-схема — шейлоны/темы/модули.** Текущая схема линейна (`topics/lessons/exercises/progress`)
  и не отражает иерархию шейлон→тема→модуль и граф + теги. Требует пересмотра до реализации бэкенда.
  Конкретный вопрос: хранение поля `order` для тем и модулей — отдельная колонка на таблице `topics`
  или отдельная таблица связей `shalon_themes`? Не финализировано.
- **Слой навигации лабиринта.** Архитектурно намечен (раздел 7), не реализован.
